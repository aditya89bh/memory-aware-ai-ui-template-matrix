# Why Remembered This

## Problem

Users may see a memory and wonder why the product saved it. Without an explanation, memory can feel arbitrary or invasive.

## When to use

Use this pattern anywhere memory is displayed, especially in drawers, cards, managers, correction flows, and consent prompts.

## UI pattern

Provide a "Why remembered?" explanation that includes:

- Source event or behavior
- Capture method such as confirmed, inferred, imported, or corrected
- Date remembered
- Scope
- Reason it may be useful
- Link to edit or forget

Keep the explanation short by default with an option to expand.

## User controls

- View source
- Edit memory
- Forget memory
- Change scope
- Stop remembering similar items
- Report wrong reason

## Edge cases

- The memory was synthesized from several events.
- The original source is unavailable.
- The source contains private content.
- The reason is based on inference.
- The user disagrees with the product's usefulness claim.

## Trust risk

The main risk is mysterious persistence. Users may tolerate memory better when they understand its origin and purpose.

## Example product context

A note-taking product remembers "Prefers action items at the top" and explains that the user confirmed this preference after editing three meeting summaries.
