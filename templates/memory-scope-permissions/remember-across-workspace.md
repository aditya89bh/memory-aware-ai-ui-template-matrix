# Remember Across Workspace

## Problem

A memory is useful to a whole workspace, but applying it broadly can affect multiple people and workflows. The product needs a deliberate approval and visibility pattern.

## When to use

Use this pattern for team norms, project standards, customer account facts, brand constraints, shared workflow rules, or organization-wide preferences.

## UI pattern

Show a workspace memory proposal with:

- Memory statement
- Workspace scope label
- Who can see it
- Who can edit it
- Source and proposer
- Approval or confirmation action
- Review path

For high-impact memory, require permission checks or review.

## User controls

- Save to workspace
- Limit to project
- Keep personal
- Request approval
- Edit wording
- View workspace memory list

## Edge cases

- The user lacks permission to save workspace memory.
- The memory reveals private individual behavior.
- A team-level memory conflicts with personal preferences.
- Workspace memory should apply only to one client or project.
- The memory needs periodic review by an owner.

## Trust risk

The main risk is shared overreach. A memory created by one person can unintentionally shape everyone else's experience.

## Example product context

A design operations workspace saves "Use accessibility review before handoff" as a workspace-level workflow memory visible to the full product team.
