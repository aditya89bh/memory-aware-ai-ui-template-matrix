# Contributing

Thank you for considering a contribution. This repository is intended to become a high-quality open design reference for memory-aware AI products.

Contributions should improve the clarity, usefulness, and practical applicability of the template matrix.

## What belongs here

Good contributions include:

- New memory design templates
- Practical product examples
- Taxonomy improvements
- Design principles and review checklists
- Research questions and evaluation rubrics
- Better terminology for user-facing memory behavior
- Clear diagrams or structured artifacts that support design decisions

This repository is not intended for framework-specific UI components, backend memory storage implementations, vendor comparisons, or product marketing copy.

## Contribution standards

Before opening a pull request, check that your contribution is:

- Clear enough for designers, researchers, product managers, and builders to use
- Specific about memory behavior, user control, and trust implications
- Written in consistent terminology from `docs/taxonomy.md`
- Practical rather than abstract
- Free of duplicated content
- Respectful of privacy, consent, and user agency
- Not tied unnecessarily to a specific implementation stack

## Writing style

Use plain, professional language. Prefer concrete product wording over vague claims.

Prefer:

- "Let users review memories before they are applied."
- "Show the source of a project memory."
- "Allow users to forget a single preference."

Avoid:

- "Create a magical personalized experience."
- "Use intelligent context everywhere."
- "Make memory seamless" without explaining the user controls.

## Template contributions

When adding a template, include:

- Template name
- User problem
- When to use
- When not to use
- Memory objects involved
- Source and scope rules
- Primary user actions
- Interaction flow
- Content guidance
- Trust and safety considerations
- Accessibility considerations
- Failure modes
- Research questions
- Example product contexts

## Example contributions

When adding an example, include:

- Scenario title
- Product context
- User goal
- Memory involved
- Relevant template category
- Interaction walkthrough
- What the user can inspect
- What the user can change
- How forgetting works
- Risks and edge cases
- Success criteria

## Pull request checklist

Before submitting, confirm:

- The contribution has a clear purpose.
- Terminology is consistent with the taxonomy.
- The content adds new value rather than repeating existing guidance.
- Any sensitive memory scenario is handled carefully.
- The writing is concise and professional.
- Links, headings, and file paths are correct.

## Review philosophy

Reviews should improve the usefulness of the repository. Feedback should focus on clarity, practical application, consistency, and trustworthiness. If a contribution introduces a new concept, it should also explain how product teams can apply it.
