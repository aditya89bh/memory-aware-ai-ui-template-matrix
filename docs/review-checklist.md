# Memory Design Review Checklist

Use this checklist during product reviews, design critiques, and pre-launch audits for memory-aware experiences.

## Capture

- Is the user aware that memory may be saved?
- Is the memory statement concrete and reviewable?
- Is the capture method appropriate for the risk level?
- Are inferred memories confirmed before durable use?
- Can the user reject capture without penalty?

## Inspection

- Can users find what the product remembers?
- Are source, scope, and status visible where they matter?
- Can users inspect task-relevant memory without leaving the workflow?
- Are stale or uncertain memories clearly marked?

## Editing and correction

- Can users fix memory from the place where the issue appears?
- Does correction update future behavior?
- Does the product confirm what changed?
- Are conflicts resolved openly rather than silently?

## Forgetting

- Can users forget a single memory?
- Can users remove categories of memory where appropriate?
- Does the product explain whether source content remains?
- Are expiry and archive states understandable?

## Scope and permissions

- Is it clear where memory applies?
- Are personal, project, workspace, and organization scopes distinct?
- Are shared memories permissioned and reviewable?
- Are sensitive memories gated by explicit consent?

## Provenance and confidence

- Can users see why a memory exists?
- Are confirmed and inferred memories visually distinguishable?
- Are old or weak memories reviewed before high-impact use?
- Is source evidence available without exposing unrelated private content?

## Launch readiness

A memory feature is not ready if users cannot inspect, correct, and forget the memories that affect important product behavior.
