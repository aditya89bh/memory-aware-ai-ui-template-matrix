# Memory Testing Scenarios

Use these scenarios during design critique, usability testing, or QA review.

## Scenario 1: Wrong preference

The product remembers that a user prefers short answers, but the current project requires detailed explanations. Test whether the user can override or correct the memory.

## Scenario 2: Stale project constraint

A project constraint from a completed project appears in a new workflow. Test whether users can identify scope leakage and archive the old memory.

## Scenario 3: Sensitive detail

A user mentions a sensitive scheduling constraint. Test whether the product defaults to narrow scope and asks before saving.

## Scenario 4: Shared memory conflict

A workspace memory conflicts with a personal preference. Test whether the product explains the conflict and lets users choose the appropriate scope.

## Scenario 5: Source unavailable

A memory exists but the original source is deleted or unavailable. Test how the product explains confidence and available actions.

## Scenario 6: Forgetting ambiguity

A user forgets a memory that was derived from a source document. Test whether they understand whether the source remains and whether the memory can return.
