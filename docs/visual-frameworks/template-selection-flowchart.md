# Template Selection Flowchart

## Purpose

Turn a memory design problem into a concrete set of templates from the matrix.

## When to use

Use this when a team knows memory is involved but does not know which templates to apply.

## Diagram

```text
Start with user need
|
|-- Approve memory creation -> Memory Capture templates
|-- See what memory exists -> Memory Inspection templates
|-- Change memory -> Memory Editing templates
|-- Fix wrong memory -> Memory Correction templates
|-- Remove memory -> Memory Forgetting templates
|-- Limit where memory applies -> Scope and Permission templates
|-- Understand source or confidence -> Provenance and Confidence templates
```

## How product teams should apply it

Select one primary user need, then add supporting templates for correction and forgetting. A complete workflow usually needs at least three template families.

## Common mistakes

- Selecting only an inspection pattern and calling the system transparent.
- Forgetting correction and forgetting controls.
- Choosing templates by screen type instead of user need.
- Ignoring provenance when memory affects important outcomes.
