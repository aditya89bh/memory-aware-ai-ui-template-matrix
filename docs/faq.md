# FAQ

## Is this a component library?

No. This is a design and UX strategy repository. It helps teams decide what memory experiences should do before they choose implementation details.

## Who should use this repository?

Product designers, UX researchers, product managers, AI application builders, consultants, and research teams working on memory-aware products.

## Where should I start?

Start with [How to Use This Repository in 30 Minutes](how-to-use-in-30-minutes.md), then use the [Template Selection Guide](template-selection-guide.md) and [Memory Flow Audit Worksheet](../worksheets/memory-flow-audit.md).

## What problem does this repository solve?

It helps teams design memory that users can inspect, edit, correct, scope, and forget.

## Does every product need every template?

No. Use the [Memory Risk Rubric](memory-risk-rubric.md) and [Template Selection Guide](template-selection-guide.md) to choose only the patterns needed for your flow.

## How should teams handle sensitive memory?

Use explicit consent, narrow scope, source visibility, conservative reuse, and clear forgetting controls. See [Sensitive Memory Consent Gate](../templates/memory-scope-permissions/sensitive-memory-consent-gate.md).

## How do I know if a memory feature is launch-ready?

Use the [Design Review Scorecard](design-review-scorecard.md), [Release Checklist](release-checklist.md), and [Final Quality Audit](final-quality-audit.md).

## Can these templates be used in design tools?

Yes. See [Figma-Ready Specs](figma-ready-specs.md) for layout guidance.
